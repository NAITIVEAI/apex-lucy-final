# Lucy — Project Overview & Project Definition Record (PDR)

This document is the canonical entry point for understanding *what* Lucy is, *who* it serves, and *which* boundaries the implementation must respect. It is intentionally concise — every claim links to a deeper document under `docs/`.

---

## Mission & business context

Lucy is a member-support AI assistant for class-action settlements. Settlement-class members reach Lucy directly through a chat interface, ask questions about their notice or claim, and receive answers grounded in the official notice corpus and case-management records (Microsoft Dynamics 365). When a question is sensitive, ambiguous, or requires human authority, Lucy hands the conversation off to a human support agent through a dedicated handoff portal that bridges to Microsoft Teams.

The product replaces a high-volume manual support workload with a grounded, traceable, and auditable assistant whose answers can be reviewed, escalated, and corrected by human agents in real time.

## Users

Lucy has two distinct user populations:

1. **Settlement class members** — external end users who interact with the Chainlit-powered chat experience served by the agent service. They are not authenticated as employees; member-side authentication and verification flows are described in [`docs/architecture/authentication-architecture.md`](architecture/authentication-architecture.md).
2. **Internal support agents** — Lucy program staff who pick up handed-off conversations through the portal service. They authenticate with a shared API token and receive Teams adaptive-card notifications when escalation is required. Their day-to-day workflow is documented in [`docs/portal-guide/portal-user-guide.md`](portal-guide/portal-user-guide.md).

## Two services at a glance

The repository is a Python monorepo containing two cooperating services:

| Service | Path | Port(s) | Role |
|---|---|---|---|
| Agent | `agent/app/` | 8000 (Chainlit), 8080 (health) | Member-facing conversational AI built on Chainlit + Azure AI Foundry |
| Portal | `portal/app/` | 8001 | Internal handoff/admin portal built on FastAPI + Jinja2 + WebSockets |

Both services are deployed as independent Docker images (Python 3.12 slim base) and communicate at runtime through a shared Azure Storage account and authenticated HTTP/WebSocket calls. A condensed view of how they fit together lives in [`docs/system-architecture.md`](system-architecture.md); the full deep-dive is in [`docs/architecture/architecture-overview.md`](architecture/architecture-overview.md).

## Capabilities (in scope)

Lucy is in scope for:

- Answering settlement-notice questions grounded in the curated `lucy-notices-v2` Azure AI Search index (vector + semantic hybrid retrieval).
- Looking up class-member status from Dynamics 365 via OAuth client-credentials.
- Streaming responses generated by the Foundry v2 runtime (model deployment `gpt-5.2`, agent name `lucy`).
- Initiating real-time human handoffs through the portal, including Teams adaptive-card notifications and bidirectional WebSocket relay.
- Logging conversations, callbacks, and agent registrations through Azure Storage Tables (with in-memory fallback).
- Emitting telemetry to Azure Monitor through OpenTelemetry instrumentation.

Detailed capability descriptions live in [`docs/executive/system-capabilities-guide.md`](executive/system-capabilities-guide.md).

## Limitations (out of scope)

The current refactor does **not** cover, and the codebase intentionally avoids:

- Bank file ingestion (ACH, wire, return/bounce, nightly bank zip).
- Business Central activation.
- Generalized payout calculator logic beyond what is already present.
- CRM migration projects, model-routing redesigns, or Lucy architecture redesigns.
- Broad anti-abuse / rate-limiting expansion.
- Azure governance, eval, or red-team expansion beyond existing instrumentation.

These constraints come directly from `AGENTS.md` and apply to all contributors. Any work that drifts into these categories is rejected at review time.

## Stack overview

- **Language & runtime**: Python 3.12 (async-first; asyncio + aiohttp + Chainlit message handlers).
- **Agent**: Chainlit 2.9.5 with FastAPI 0.116.1 / Uvicorn 0.35.0 under the hood.
- **Portal**: FastAPI 0.116.1 + Jinja2 templates + vanilla-JS frontend + WebSockets.
- **AI**: Azure AI Foundry (gpt-5.2 deployment), Azure AI Agents 1.x, Azure AI Projects 2.x.
- **Retrieval**: Azure AI Search index `lucy-notices-v2` with `vector_semantic_hybrid` query type.
- **CRM**: Microsoft Dynamics 365 via OAuth client-credentials.
- **Collaboration / handoff**: Microsoft Teams Graph (adaptive cards) + portal WebSockets.
- **Storage**: Azure Storage Tables (registry, conversation, callbacks) + Azure Blob Storage for artifacts.
- **Observability**: OpenTelemetry → Azure Monitor (`tracing_config.py`, `real_metrics_system.py`).

For integration-level depth see [`docs/integrations/azure-search-integration.md`](integrations/azure-search-integration.md), [`docs/integrations/dynamics365-integration.md`](integrations/dynamics365-integration.md), and [`docs/integrations/teams-integration.md`](integrations/teams-integration.md).

## Workflow note — spec-driven, ledger-tracked

This repository operates under a **strict spec-driven workflow** defined in [`AGENTS.md`](../AGENTS.md). Contributors and AI agents must:

1. Read `state/refactor-ledger.md` first.
2. Pick the lowest-numbered incomplete plan in `plans/` (read **exactly one** active plan).
3. Use the source-of-truth hierarchy: explicit user instruction → `AGENTS.md` → active plan → existing code patterns.
4. Update the ledger after every completed or blocked plan.

Currently `plans/` and `state/refactor-ledger.md` are uninitialized; the workflow becomes binding the moment plans are added. Do not start parallel write-heavy work that crosses plan boundaries. Do not "improve adjacent things."

The full operating contract — required subagents, scope guardrails, validation rules, blocker handling, plan-specific reminders for plans 001–005 — is in `AGENTS.md`. This file does not restate it; it points to it.
